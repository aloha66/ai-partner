# AI 桌面伴侣助手业务需求说明（精简版）

> 面向对象：产品、客户成功、销售、合作伙伴沟通
> 文档目的：说明 AI 桌面伴侣助手的核心业务需求、用户可见体验与对外表达边界

## 1. 产品定位

AI 桌面伴侣助手是一款轻量桌面陪伴应用。它在用户电脑桌面展示一个可互动的动态伴侣，并根据 AI 编码助手的工作状态切换动作和提示，让用户更直观地感知 AI 当前正在做什么。

它的核心价值不是“在桌面放一张图片”，而是让伴侣成为用户和 AI 编码助手之间的可视化反馈层：

- 当 AI 助手开始阅读、搜索、编辑或运行任务时，伴侣会出现对应动作。
- 当任务完成、等待用户授权、发生错误或会话结束时，伴侣会用不同状态提醒用户。
- 用户可以在桌面上拖动伴侣、切换本地伴侣、暂停联动或关闭伴侣。
- 伴侣以轻量方式常驻，不替代编辑器、终端或 AI 工具本身。

一句话介绍：

> AI 桌面伴侣助手是一个会响应 AI 工作流的桌面伴侣，让用户在使用 Codex、Claude Code、Gemini、OpenCode、Antigravity 等工具时获得更直观、更有陪伴感的状态反馈。

## 2. 目标用户与适用场景

### 2.1 个人 AI 编程用户

适合已经在日常工作中使用 AI 编码助手的个人用户。

用户可以：

- 在桌面显示动态伴侣。
- 让伴侣跟随 AI 助手的工作状态做出反馈。
- 在工作中通过伴侣气泡看到简短提示，例如正在读取内容、正在编辑、正在等待确认等。
- 在需要专注时暂停桌面伴侣联动，并在之后恢复。
- 在本地可用伴侣中切换自己想展示的形象。

### 2.2 AI 工具使用场景

AI 桌面伴侣助手可以作为 AI 工具的轻量反馈界面。

适合对外表达为：

- 给 AI 编码助手增加更友好的状态反馈。
- 将任务执行过程转化为可感知的桌面陪伴体验。
- 帮助用户更容易发现“AI 正在工作、等待我、完成了、出错了”等状态。

## 3. 核心功能总览

| 功能模块 | 用户能获得什么 |
| --- | --- |
| 桌面伴侣展示 | 在桌面常驻显示动态伴侣，不打断当前工作 |
| 伴侣动作反馈 | 伴侣会根据 AI 助手工作状态切换动作 |
| 气泡提示 | 用简短文字说明当前 AI 助手正在做什么 |
| 本地伴侣选择器 | 用户可以在本地伴侣资产中搜索、切换和管理 |
| 多 AI 助手支持 | 支持 Codex、Claude Code、Gemini、OpenCode、Antigravity 等使用场景 |
| 陪伴模式 | 用户可随时主动进入高互动陪伴状态，不只在闲置时触发 |
| 深度触摸互动 | 触摸伴侣不同部位会触发不同动作、反馈与关系变化 |
| 性格养成与情绪同步 | 伴侣可根据用户互动、工作节奏和长期偏好形成更稳定的陪伴风格 |
| 私人秘书与记忆 | 可选接入第三方大模型，提供个人助理、独立记忆和项目记忆能力 |
| 一键开关 | 用户可以开启、暂停、恢复桌面伴侣和 AI 状态联动 |
| 安全与隐私边界 | 本地联动有访问控制，不把代码内容作为产品能力采集或解释 |
| 稳定运行机制 | 状态防抖、自愈和失败降级减少对用户工作流的干扰 |

## 4. 用户可见体验

### 4.1 桌面浮动伴侣

用户会在桌面看到一个小型动态伴侣。它默认保持轻量存在，不抢占窗口焦点，也不会像普通应用窗口一样打断用户操作。

用户可以：

- 拖动伴侣到合适位置。
- 看到伴侣在不同工作状态下切换动作。
- 通过右键菜单打开本地伴侣选择器。
- 在需要时暂停、恢复或退出桌面伴侣。

对客户沟通时可以强调：

> 它是一个“工作陪伴层”，不是一个需要用户频繁操作的新工作台。

### 4.2 伴侣动作状态

伴侣会用动作表达不同状态。由于陪伴模式会承载更丰富的趣味动作，基础工作流状态应保持克制。业务上可以理解为 7 类基础状态：

| 状态 | 用户感知 |
| --- | --- |
| 待机 | 当前没有明显任务，伴侣保持默认状态 |
| 打招呼 | 任务完成或会话结束 |
| 失败 | 任务遇到错误 |
| 等待 | AI 助手正在等待用户确认、授权或输入 |
| 通用运行中 | AI 助手正在执行普通任务 |
| 阅读/审查 | AI 助手正在读取、搜索、检查或审阅内容 |
| 被用户拖动/拎起来 | 用户正在移动伴侣，伴侣表现出被拿起、悬空或跟随拖拽的反馈 |

这套状态让用户不用盯着终端或日志，也能感知 AI 助手的节奏。

### 4.3 气泡提示

桌面伴侣上方会出现短文字气泡，用来解释当前发生的事情。

示例表达包括：

- 正在读取项目内容。
- 已完成一次编辑。
- 正在运行某个工具。
- 正在搜索某段内容。
- 正在等待用户操作。

气泡设计原则：

- 文案简短，不制造额外阅读负担。
- 只显示与当前动作相关的信息。
- 配合不同 AI 助手显示对应来源标识，方便用户知道是哪一个工具触发了提示。
- 当伴侣选择器打开时，界面会自动让出空间，避免内容互相遮挡。

### 4.4 本地伴侣选择与管理

用户可以通过右键打开伴侣选择器，查看本地可用伴侣。

选择器支持：

- 查看伴侣列表。
- 搜索伴侣。
- 切换当前桌面伴侣。
- 在伴侣数量较多时保持流畅浏览。
- 退出桌面伴侣，并有二次确认，避免误操作。

对产品沟通时可以说：

> 用户不是只能使用默认伴侣，而是可以把自己喜欢的本地伴侣形象放到桌面，并随时切换。

补充说明：

- 桌面端优先展示用户本地已有伴侣资产。
- 本地伴侣资产需要保持统一目录与元数据规范，方便后续被桌面端识别。
- 选择器只承担本地浏览、搜索和切换职责，不承载内容发现或分发职责。

### 4.5 开启、暂停与恢复

用户可以控制桌面伴侣是否参与当前工作流：

- 开启：恢复桌面伴侣展示和 AI 助手联动。
- 暂停：停止联动并关闭桌面伴侣，适合专注、演示或不想被打扰的时候。
- 切换：根据当前状态自动开启或暂停。

这套控制能力也可以在部分 AI 助手内部通过快捷入口触发，方便用户不离开当前工作流。

## 5. 陪伴模式与深度互动

陪伴模式是 AI 桌面伴侣助手从“状态反馈层”走向“桌面伙伴”的关键能力。它不是只能在用户闲置时出现的彩蛋，而是用户可以随时主动激活、随时退出的高互动模式。

进入陪伴模式后，伴侣仍然保留 AI 工作流状态反馈能力，但会增加更丰富的互动行为、情绪表达、触摸反馈和对话能力。它的定位是：

> 在不打断用户工作的前提下，让伴侣成为可以互动、可以记住偏好、可以陪用户一起完成工作的桌面伙伴。

### 5.1 陪伴模式的使用方式

用户可以在任何工作状态下进入陪伴模式，包括正在编码、正在等待 AI 执行、任务完成后、或只是想短暂互动时。

典型入口包括：

- 点击或双击桌面伴侣。
- 通过右键菜单选择“进入陪伴模式”。
- 使用快捷键或托盘入口快速切换。
- 从 AI 助手插件或状态入口中激活。
- 由用户配置在特定时间段自动建议进入，例如长时间工作后、任务完成后、休息时间。

陪伴模式需要支持清晰的退出和降级：

- 用户可以随时退出陪伴模式，回到普通状态反馈。
- 用户可以设置安静陪伴，只保留动作反馈，不主动发起闲聊。
- 用户可以设置专注时段，在该时段中减少非必要互动。
- 当出现 AI 工具授权、错误、任务完成等高优先级事件时，陪伴模式自动让位给任务通知。

### 5.2 与工作流通知并发运行

陪伴模式不会替代现有的 AI 工作流通知，而是与工作流通知并发运行。

业务规则建议如下：

| 事件类型 | 优先级 | 伴侣表现 |
| --- | --- | --- |
| 等待用户确认、授权或输入 | 最高 | 立即展示等待状态和任务气泡，暂停闲聊表达 |
| 执行失败、异常或连接中断 | 高 | 展示失败状态，优先提示用户需要关注的问题 |
| 任务完成、会话结束 | 高 | 展示完成反馈，可在完成后回到陪伴语境 |
| AI 正在读取、搜索、编辑、运行 | 中 | 伴侣保留工作状态动作，同时可维持轻量陪伴姿态 |
| 用户主动触摸、对话或互动 | 中 | 如果没有更高优先级任务通知，则立即响应用户 |
| 伴侣主动闲聊、情绪表达、陪伴动作 | 低 | 仅在不干扰工作流通知时出现 |

对外表达重点：

> 陪伴模式让伴侣更像伙伴，但任务弹窗和关键工作流提醒始终拥有更高优先级。

### 5.3 物理触摸与动态反馈

进入陪伴模式后，用户可以像和真实桌面伴侣互动一样，通过鼠标、触控板或触摸屏与伴侣进行局部互动。

支持的互动方式包括：

- 点击：触发轻量反应，例如眨眼、点头、打招呼。
- 双击：触发更明显的动作，例如跳跃、转身、撒娇、展示状态。
- 长按：触发持续反馈，例如被摸头、抱住、害羞、放松。
- 拖拽：伴侣跟随移动，并根据拖拽速度或释放位置做出反应。
- 悬停：伴侣看向指针、呼吸变化、轻微表情变化。
- 连续互动：根据短时间内的互动频率产生不同反应，例如开心、疑惑、抗议或需要休息。

不同部位可以有不同反馈。示例包括：

| 触摸部位 | 可能反馈 |
| --- | --- |
| 头部 | 摸头、开心、眯眼、亲近度提升 |
| 脸部 | 眨眼、害羞、歪头、打招呼 |
| 身体 | 放松、伸展、坐下、陪伴姿态 |
| 手/爪/翅膀 | 击掌、挥手、跳跃、工具启动反馈 |
| 尾巴/耳朵/装饰物 | 惊讶、躲闪、甩尾、特殊动作 |
| 道具或挂件 | 打开快捷菜单、触发伴侣专属能力或主题动作 |

每个伴侣都应该能够自定义可触摸部位，而不是由产品写死。伴侣资产可以通过元数据描述：

- 可触摸区域名称。
- 区域形状或命中范围。
- 默认动作与特殊动作。
- 对情绪、亲密度或性格的影响。
- 是否允许用户自定义该区域的反馈。
- 是否绑定特定能力，例如打开提醒、记录想法、暂停联动。

这样可以支持不同伴侣拥有不同交互个性。例如猫型伴侣可以更强调摸头、尾巴和撒娇反馈；机器人伴侣可以强调按钮、屏幕、模块化工具反馈；幻想类伴侣可以强调翅膀、角、饰品和特殊动作。

### 5.4 性格养成与情绪同步

伴侣不应只是随机播放动作，而应该在长期互动中形成可感知的性格和关系变化。

性格养成可以包括：

- 基础性格：活泼、安静、毒舌、温柔、认真、好奇等。
- 互动偏好：喜欢被摸头、喜欢被夸奖、讨厌高频打扰、喜欢在任务完成时庆祝等。
- 关系阶段：陌生、熟悉、亲近、默契等，用来影响动作和语气。
- 工作陪伴风格：鼓励型、秘书型、吐槽型、安静守护型、效率教练型等。
- 成长记录：根据用户长期使用、完成任务、互动频率形成轻量成长反馈。

情绪同步可以基于用户主动设置和本地可见的工作节奏进行，不应默认推断用户隐私或敏感心理状态。

可支持的情绪来源包括：

- 用户主动选择当前状态，例如专注、疲惫、开心、焦虑、休息中。
- 工作流状态，例如长时间运行、频繁失败、连续完成任务、长时间无互动。
- 用户与伴侣的互动，例如频繁触摸、主动聊天、忽略提醒。
- 用户配置的时间和节奏，例如工作日、休息时间、深夜、会议时段。

伴侣反馈可以包括：

- 根据用户状态调整语气和动作强度。
- 在长时间工作后建议休息，但不强制打断。
- 在任务失败后给出安抚或鼓励。
- 在连续完成任务后庆祝或记录成就。
- 在用户选择安静模式时降低主动表达。

### 5.5 可选大模型能力与私人秘书

陪伴模式可以提供可选的第三方大模型接入，让伴侣从“会动的状态反馈”进一步变成个人私人秘书。

这项能力应明确为可选增强能力，而不是基础联动的默认前提。用户不接入大模型时，仍然可以使用桌面伴侣展示、状态联动、触摸互动和本地陪伴能力。

接入大模型后，伴侣可以提供：

- 自然语言闲聊和陪伴对话。
- 当前任务的简短解释与提醒。
- 工作节奏建议，例如什么时候休息、什么时候回看待办。
- 个人偏好记忆，例如称呼、喜欢的语气、常用工作方式。
- 项目上下文记忆，例如当前项目目标、常见命令、近期任务、未完成事项。
- 私人秘书能力，例如提醒事项、会议前提示、任务清单整理、阶段性总结。
- 多项目切换时的上下文区分，避免把不同项目的记忆混在一起。

记忆能力建议拆分为两类：

| 记忆类型 | 说明 |
| --- | --- |
| 伴侣独立记忆 | 与伴侣自身相关，例如性格、亲密度、互动偏好、用户称呼、语气偏好 |
| 当前项目记忆 | 与工作项目相关，例如项目目标、常用入口、近期任务、用户明确保存的上下文 |

对外表达重点：

> 用户可以选择让伴侣成为更懂自己的私人秘书，但记忆与大模型能力必须由用户主动开启、可查看、可编辑、可清除。

### 5.6 陪伴模式的边界

陪伴模式需要保持“陪伴但不越界”的产品边界。

建议边界包括：

- 不默认读取用户屏幕、聊天记录、代码内容或文件内容。
- 不把用户情绪识别表达为医学、心理诊断或自动判断。
- 不在用户专注或演示场景中主动弹出大量闲聊。
- 不让伴侣回复覆盖高优先级任务提醒。
- 不把第三方大模型接入包装成默认能力，必须说明需要用户授权和配置。
- 不让项目记忆跨项目、跨工作区或跨账号混用。

推荐话术：

> 陪伴模式让桌面伴侣更有生命感，但它仍然尊重用户的工作节奏、隐私授权和任务优先级。

## 6. AI 助手联动能力

AI 桌面伴侣助手当前面向多种 AI 编码助手提供联动体验。

重点覆盖的使用场景包括：

- Claude Code
- Codex
- Gemini
- OpenCode
- Antigravity

不同 AI 助手的底层接入方式可以不同，但用户看到的是统一的伴侣状态和气泡体验。

用户在这些工具中工作时，AI 桌面伴侣助手可以根据常见事件展示伴侣状态：

| AI 助手行为 | 桌面伴侣反馈 |
| --- | --- |
| 开始执行工具 | 伴侣进入运行状态 |
| 读取、搜索、检查内容 | 伴侣进入阅读/审查状态 |
| 工具执行结束 | 伴侣回到待机 |
| 会话结束 | 伴侣打招呼 |
| 会话出错 | 伴侣显示失败状态 |
| 等待用户确认 | 伴侣显示等待状态 |
| 用户提交新请求 | 伴侣进入通用运行中状态 |

产品表达重点：

> AI 桌面伴侣助手让 AI 助手的后台行为变成用户可见的轻量反馈，降低“它现在到底在干什么”的不确定感。

## 7. 稳定性与隐私边界

### 7.1 稳定性要求

AI 桌面伴侣助手的联动设计目标是“有反馈，但不打扰”。业务沟通中可以强调：

- 高频事件会做节奏控制，避免伴侣状态频繁闪烁。
- 连续运行状态会有轻微动作变化，让反馈更自然。
- 桌面伴侣异常不应阻断用户正在使用的 AI 工具。
- 连接异常时应尽量降级为安静状态，避免把桌面伴侣问题放大成主工作流问题。

这部分适合支持“不打断工作流”和“适合长期陪伴”的产品叙事。

### 7.2 安全与隐私边界

AI 桌面伴侣助手会在本机接收 AI 助手状态并驱动桌面伴侣。对外沟通时建议明确：

- 本地服务只面向本机使用，不设计为远程开放控制入口。
- 状态和气泡等写入操作需要本机访问控制，避免外部来源随意触发。
- 气泡文案来自确定性的工具事件模板，不是让桌面伴侣读取项目后用 AI 自行总结。
- 气泡文本和请求体有长度限制，状态类型也有白名单，避免失控插件刷屏或写入异常内容。
- 产品不应表达为会读取、上传或分析用户项目代码内容。
- 陪伴模式下的闲聊、大模型接入和记忆能力需要用户主动开启。
- 第三方大模型服务商、模型类型、数据发送范围和保存策略需要对用户可见。
- 伴侣独立记忆和项目记忆应支持查看、编辑、删除、暂停和导出。
- 项目记忆默认限定在当前项目或工作区，不应自动用于其他项目。
- 用户关闭大模型或记忆能力后，桌面伴侣仍应能以本地模式继续运行。

推荐话术：

> AI 桌面伴侣助手主要处理本机 AI 工具事件和简短状态提示，目标是展示工作流状态，而不是读取、上传或分析用户项目内容。

## 8. 平台支持

产品需要支持 macOS 和 Windows 两个平台。
